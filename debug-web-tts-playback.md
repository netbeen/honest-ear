# [OPEN] web-tts-playback

## 背景
- 现象：页面已经显示回复文案，但 TTS 控件显示 `0:00 / 0:00`，且无法播放。
- 目标：定位是 TTS 文件生成、接口返回，还是前端播放链路的问题。
- 约束：先收集运行时证据，不先改业务逻辑。

## 假设
1. 后端生成的是浏览器不支持的音频格式，导致无法解码。
2. `/v1/tts-audio` 返回了空文件、错误文件或错误响应头。
3. 前端设置 `audio.src` 和触发 `load/play` 的时机有问题。
4. TTS 文件存在，但文件路径校验或流式返回过程有问题。

## 计划
1. 检查当前服务返回的 `tts_output`、`tts_audio_url` 和音频文件实际格式。
2. 直接请求 `/v1/tts-audio`，确认响应头、文件大小和可播放性。
3. 如有必要，再补最小化插桩定位前端或接口问题。

## 当前状态
- 状态：已确认根因并完成修复，等待用户在页面中复验。

## 证据
| 假设 | 结论 | 证据 |
|---|---|---|
| 1. 后端生成的是浏览器不支持的音频格式 | 是 | `file` 显示旧文件为 `AIFF-C compressed audio`，页面表现为 `0:00 / 0:00` |
| 2. `/v1/tts-audio` 返回了空文件或错误内容 | 否 | 实际返回的是非空音频文件，文件大小约 `321K` |
| 3. 前端设置音频地址时机有问题 | 基本否 | 接口已返回 `tts_audio_url`，前端也成功设置了 `<audio>` 的 `src` |
| 4. 文件路径校验或流式返回有问题 | 否 | 真实接口可以取回音频文件内容，只是格式不适合当前浏览器播放 |

## 修复
- 将 `macOS say` 的输出从中间 `AIFF` 自动转换为浏览器更稳妥的 `WAVE PCM`
- 保留原有前端播放链路，不改交互方式

## 修复后验证
- 新生成文件已验证为：`RIFF (little-endian) data, WAVE audio, Microsoft PCM, 16 bit, mono 22050 Hz`
- 本地服务已重启，等待页面复验
