# [OPEN] process-upload-latency

## 背景
- 现象：`/v1/process-upload` 最近一次调用耗时约 32 秒。
- 目标：定位主要耗时发生在上传落盘、ASR、LLM 还是 TTS。
- 约束：先收集运行时证据，不先改业务逻辑。

## 假设
1. 双路 ASR 是主要耗时来源，某个模型推理明显慢。
2. LLM 请求是主要耗时来源，当前 `chat/completions` SSE 返回导致整体等待时间拉长。
3. 上传文件保存到临时文件阶段出现异常慢 I/O。
4. 可选 TTS 或其他串行阻塞步骤放大了总耗时。

## 计划
1. 在上传入口与 pipeline 关键阶段加入最小化耗时日志。
2. 重启服务并复现一次上传请求。
3. 根据阶段耗时判断根因，再决定是否需要优化。

## 当前状态
- 状态：已完成一次有效复现，根因已基本确认。

## 证据
| 假设 | 结论 | 证据 |
|---|---|---|
| A. 双路 ASR 是主要耗时来源 | 否 | `trae-debug-log-process-upload-latency.ndjson` 中 faithful ASR 约 `458.3ms`，intended ASR 约 `1225.4ms`，总计远小于 32 秒 |
| B. LLM 请求是主要耗时来源 | 是 | 同一日志中 LLM 耗时约 `25363.57ms`，占总时长 `26630.19ms` 的绝大部分 |
| C. 上传文件保存慢 | 否 | 上传落盘仅 `0.4ms` |
| D. 其他串行步骤放大耗时 | 基本否 | fusion `0.07ms`，pipeline 总耗时 `26621.3ms`，与 LLM 耗时几乎重合 |

## 结论
- 当前 `/v1/process-upload` 的长耗时主要由 LLM 阶段导致，不是上传落盘，也不是 ASR 预热或本地模型推理。
- 这次有效复现里，总耗时约 `26.63s`，其中 LLM 单段就用了约 `25.36s`。
- 切换到 Ark 官方 Python SDK 后，问题仍然存在，只是总耗时有所波动下降。

## 二次验证
| 场景 | 总耗时 | ASR 耗时 | LLM 耗时 | 结论 |
|---|---:|---:|---:|---|
| 旧 HTTP 路径一次有效复现 | `26630.19ms` | `458.3ms + 1225.4ms` | `25363.57ms` | 主要卡在 LLM |
| Ark SDK 路径一次有效复现 | `24659.43ms` | `308.24ms + 1232.79ms` | `23361.23ms` | 仍主要卡在 LLM |
| Ark SDK 路径另一次有效复现 | `17977.73ms` | `2060.16ms + 1380.66ms` | `15719.68ms` | 仍主要卡在 LLM |

## 当前判断
- 切到 Ark SDK 没有消除核心瓶颈，只是把一部分请求从约 `26.6s` 降到约 `24.7s`，甚至有一次降到约 `18.0s`。
- 说明瓶颈更像是“模型侧生成/排队/推理时间波动”，而不是我们客户端协议栈的固定额外开销。

## 下一步
1. 如需继续优化，应进一步拆分 LLM 阶段，记录“请求发出时间 / 首个 SSE chunk 到达时间 / 最后一个 chunk 完成时间”。
2. 若要快速降时延，可优先尝试更快的模型、缩短 prompt、减少输出长度，或切回更低延迟的后端。
